# Product

## Register

brand

## Users

Backend engineers, Rust developers, Rinha de Backend participants, and technical reviewers who land on the GitHub Pages site from the repository, challenge community links, CI reports, or search. They are usually evaluating implementation credibility quickly: what was built, how it performs, what tradeoffs were made, and whether the repository is worth reading, cloning, benchmarking, or comparing against another implementation.

## Product Purpose

This site is the public technical showcase for `rinha2-back-end-rust`, a Rust/Actix-web implementation of Rinha de Backend 2024/Q1. It should communicate engineering seriousness, extreme resource constraints, performance evidence, and navigable documentation without becoming a generic SaaS landing page. Success means a technical visitor can understand the project promise in the first viewport, trust the benchmark claims, reach docs or GitHub quickly, and inspect historical stress test artifacts without friction.

## Brand Personality

Industrial, compressed, and evidence-driven. The voice can be playful about Rust compilation, but the presentation should feel like a benchmark rig, not a meme page. Copy should be terse, technical, and specific; proof should do more work than adjectives.

## Anti-references

- Generic AI landing pages with gradient text, glowing cards, identical metric grids, and vague speed claims.
- Rust/terminal cosplay where monospace is used everywhere without improving comprehension.
- Corporate SaaS polish that hides the challenge constraints behind marketing fluff.
- Dashboard overload that turns the homepage into a monitoring app rather than a curated project showcase.
- Decorative cyberpunk effects that compete with the benchmark evidence.

## Design Principles

1. Lead with proof, not hype: performance claims need visible context, provenance, and links to source artifacts.
2. Preserve the Rust-industrial mood: dark brown/black, oxidized orange, warm metal, and constrained mechanical details are part of the identity.
3. Make constraints legible: CPU, memory, topology, and challenge rules should read as the central drama of the page.
4. Give every section a job: hero for positioning, architecture for comprehension, stress tests for trust, reports for auditability.
5. Prefer engineered composition over generic cards: use rails, diagrams, timelines, terminals, gauges, and compact data modules where they clarify the system.

## Accessibility & Inclusion

Target WCAG AA for text contrast, focus states, and keyboard navigation. Support `prefers-reduced-motion`, avoid relying on orange alone to encode meaning, and keep long explanatory copy in readable proportional text rather than all-monospace. Motion should feel like machinery warming up, but it must never be required to understand or navigate the page.
